how it works

Your codebase becomes a graph. Changes become queries.

Parsers emit a language-agnostic intermediate representation. The graph builder turns it into a dependency graph. The engine traverses it to compute impact.

what you can ask

Six questions, answered before you merge.

what breaks if I change this?

Traverse CALLS and DEPENDS_ON edges to find every affected function — direct callers and everything downstream.

which files depend on it?

Module and package nodes roll transitive dependencies up to the file level, so you see the surface area at a glance.

which endpoints are hit?

EXPOSES edges connect functions to API endpoints, so you know exactly what your users can feel.

which tests should run?

TESTS edges map coverage back to code. Run the affected subset instead of the whole suite.

how risky is this change?

A weighted score over callers, endpoints, tests, and call depth — one number for the whole PR.

who should review it?

The AI layer reads the graph context — diff, dependencies, tests — and suggests reviewers and migration plans.

risk score

One number per change.

Computed from the graph, not from vibes. Future versions add graph centrality, change history, and coverage.

risk = 0.4 × callers // who calls this, transitively + 0.3 × api_endpoints // user-facing surface + 0.2 × tests // coverage touching the change + 0.1 × call_depth // how deep the ripple goes
